# OIDC + Changesets Publish System Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace NPM_TOKEN-based publishing with OIDC token-less auth, integrate @changesets/cli for versioning + changelog generation, and auto-create GitHub Releases on publish.

**Architecture:** Changesets manages versioning and changelogs. On push to main, `changesets/action` either opens a "Version Packages" PR (if pending changesets exist) or publishes + creates a GitHub Release (if the Version PR was just merged). npm authentication uses OIDC trusted publishers — no secrets needed. A sync script copies the generated CHANGELOG.md from `npm/` to the project root (keeping `npm/CHANGELOG.md` in place so changesets can append to it on subsequent runs).

**Tech Stack:** @changesets/cli, @changesets/changelog-github, changesets/action (GitHub Action), npm OIDC trusted publishers

**Design doc:** `docs/plans/2026-03-19-oidc-changeset-publish-design.md`

**Files:**

- [ ] **Step 1: Add new scripts to root package.json**

Add these scripts (keep existing ones):

```json
{
  "scripts": {
    "build:wasm": "bash scripts/build.sh",
    "build:js": "rolldown -c && tsc --emitDeclarationOnly",
    "build": "bun run build:wasm && bun run build:js",
    "changeset": "changeset",
    "version": "changeset version && node scripts/sync-changelog.js && git add CHANGELOG.md",
    "release": "bun run build && cd npm && npm publish --provenance --access public"
  }
}
```

Script purposes:
- `changeset`: Shortcut for `bunx changeset` (add changesets interactively)
- `version`: Runs `changeset version` (bumps version + generates changelog in `npm/`), then syncs CHANGELOG.md to root, then stages it so `changesets/action` includes it in the version commit
- `release`: Full build + publish with OIDC provenance

Note: The `git add CHANGELOG.md` at the end of `version` is important — `changesets/action` auto-commits all staged changes after running the version command, but it only knows about files that changesets itself modified (`npm/package.json`, removed changeset files). The root `CHANGELOG.md` is created by our sync script, so we must explicitly stage it.

### Task 5: Create CHANGELOG.md sync script

**Files:**
- Create: `scripts/sync-changelog.js`

Changesets generates CHANGELOG.md inside the package directory (`npm/`). We want a copy at the project root. This script copies (not moves) `npm/CHANGELOG.md` to root — keeping the original in place so changesets can append to it on subsequent version bumps.

- [ ] **Step 1: Create `scripts/sync-changelog.js`**

```js
import { existsSync, copyFileSync } from "node:fs";
import { resolve, dirname } from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = dirname(fileURLToPath(import.meta.url));
const rootDir = resolve(__dirname, "..");
const src = resolve(rootDir, "npm/CHANGELOG.md");
const dest = resolve(rootDir, "CHANGELOG.md");

if (existsSync(src)) {
  copyFileSync(src, dest);
  console.log("Copied npm/CHANGELOG.md → CHANGELOG.md");
} else {
  console.log("No npm/CHANGELOG.md found, skipping sync");
}
```

Why `copyFileSync` instead of `renameSync`: changesets needs `npm/CHANGELOG.md` to exist on subsequent runs so it can append new entries at the top. If we moved (renamed) it, the next `changeset version` would create a fresh file with only the latest entry, losing all prior history. By copying, `npm/CHANGELOG.md` stays in place (gitignored) and the root copy is the tracked version.


Expected: `No npm/CHANGELOG.md found, skipping sync` (since no changelog exists yet)

### Task 6: Replace GitHub Actions release workflow

**Files:**
- Replace: `.github/workflows/release.yml`

This is the core change: from tag-triggered + NPM_TOKEN to main-branch-triggered + OIDC + changesets/action.

- [ ] **Step 1: Replace `.github/workflows/release.yml`**

```yaml
name: Release

on:
  push:
    branches:
      - main

permissions:
  contents: write
  pull-requests: write
  id-token: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - uses: actions/setup-go@v5
        with:
          go-version: "1.26"

      - uses: actions/setup-node@v4
        with:
          node-version: "22"
          registry-url: "https://registry.npmjs.org"

      - uses: oven-sh/setup-bun@v2

      - name: Install dependencies
        run: bun install

      - name: Create Release Pull Request or Publish
        uses: changesets/action@v1
        with:
          version: bun run version
          publish: bun run release
          createGithubReleases: true
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Key differences from the old workflow:
- **Trigger:** `push: branches: [main]` instead of `push: tags: [v*]`
- **Permissions:** `contents: write` + `pull-requests: write` + `id-token: write`
- **No NPM_TOKEN:** OIDC handles authentication via `id-token: write` permission. The `setup-node` action with `registry-url` configures `.npmrc`, and `npm publish --provenance` handles the OIDC token exchange automatically when trusted publishing is configured on npmjs.com.
- **changesets/action:** Handles the version-or-publish decision automatically
- **createGithubReleases:** Automatically creates GitHub Releases with changelog content

Note on OIDC auth: `setup-node` with `registry-url` writes an `.npmrc` referencing `$NODE_AUTH_TOKEN`. With npm's OIDC trusted publishing, `npm publish --provenance` performs the OIDC token exchange and does not require `NODE_AUTH_TOKEN` to be set. If the first publish fails with an auth error, add `NODE_AUTH_TOKEN: ""` to the env block as a workaround.
